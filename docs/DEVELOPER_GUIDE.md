# 🛠️ Aether Developer Guide: Building for the Swarm

Welcome to the definitive guide for developing within the Aether ecosystem. Aether is not a chatbot; it is a high-bandwidth, autonomous execution engine built for scale, persistence, and self-evolution.

---

## 1. Best Practices: Core Patterns

To build effectively with Aether, you must master the three pillars of its architecture: **SCION**, **MCP**, and **Gas Town**.

### 🏛️ SCION (Supervisor-Controlled Intelligence & Orchestration Network)
The SCION pattern enforces a strict hierarchy between strategic reasoning and technical execution.
- **Supervisor (The Brain)**: Focuses on task decomposition, delegation, and result synthesis. It never performs technical work directly.
- **Workers (The Specialists)**: High-performance agents designed for narrow domains (e.g., Toolsmith, Auditor).
- **Rule**: Never bloat the Supervisor with domain-specific tools. Always delegate to a Worker.

### 📡 MCP (Model Context Protocol)
All communication between the Control Plane (Supervisor) and Execution Plane (Workers) happens via MCP.
- **Decoupling**: The Supervisor invokes tools and resources via a standardized interface without needing to know the implementation details.
- **Dynamic Discovery**: New capabilities generated by the Toolsmith are immediately registered and discoverable via MCP.
- **Best Practice**: Use MCP over SSE (Server-Sent Events) for stable, long-running connections in Cloud Run environments.

### ⛽ Gas Town (Continuous Persistence)
Inspired by Steve Yegge's vision, Aether operates as a persistent factory of intent.
- **Micro-Epoch Gating**: Work is divided into atomic "micro-epochs."
- **State Checkpointing**: After every epoch, the system state (history, variables, context) is serialized and persisted to Google Cloud Storage (GCS).
- **Resurrection**: If an instance is terminated, the engine "resurrects" from the latest GCS checkpoint, ensuring zero loss of momentum.
- **Best Practice**: Design your workers to be stateless; rely on the Gas Town engine to manage persistence.

## 2. Architectural Patterns

### 🕵️ Adversarial Orchestration (The Spy Game)
To eliminate single-agent bias and hallucinations, Aether employs a dialectic process known as **Adversarial Orchestration**. This is famously used in the **Spy Game** scenario where one agent (The Infiltrator/Proponent) proposes a solution while another (The Sentinel/Critic) attempts to find flaws, vulnerabilities, or logical fallacies.
- **The Dialectic**:
    1. **Proponent**: Generates a solution or code block.
    2. **Critic**: Scrutinizes the proposal and provides a structured critique.
    3. **Judge (Supervisor)**: Adjudicates the debate, forcing the Proponent to refactor until the Critic's concerns are addressed.

### 🏗️ Persistent Builds (Toolsmith Evolution)
Aether moves beyond "using tools" to "generating capabilities."
- **Pattern**: When a capability gap is identified (e.g., "I need to analyze PDF structures"), the **Toolsmith** worker synthesizes a new Python tool.
- **Persistence**: This tool is not ephemeral. It is written to the filesystem, tested via internal gates, and registered permanently in the **Agent Router**.
- **Result**: The swarm becomes more capable with every run, building its own sovereign library of specialized utilities.

## 3. Building with Aether: Step-by-Step

### Step 1: Create a New Specialist
Do not write an agent from scratch. Copy the `agents/aether_worker_template.md` and define your domain expertise.

```markdown
---
name: security_auditor
description: "Specialized in OWASP Top 10 and static analysis."
tools:
  - read_file
  - run_shell_command
model: gemini-2.0-flash
---
# Security Auditor instructions here...
```

### Step 2: Register with the Router
The **Agent Router** maps semantic intent to your specialist. You can register via the Python API or by updating `core/router_registry.json`.

**Python API:**
```python
from core.router import AgentRouter

router = AgentRouter()
router.register_tool(
    intent="Analyze code for security vulnerabilities",
    tool_name="security_auditor",
    path="agents/security_auditor.md",
    score=0.95,
    preferred_model="claude-3-5-sonnet-v2"
)
```

### Step 3: Implement Eval Gates
Every new skill must pass a **Competency Gate** before it is trusted by the Supervisor.

```python
from evals.judge_scoring import Judge

judge = Judge()
test_cases = [
    {"query": "Check this script for SQL injection", "reference": "Should identify the vulnerable execute() call."}
]

# This will trigger the Toolsmith if the score falls below the threshold (default 4.0)
passed = judge.validate_worker_competency("security_auditor", test_cases)
```
